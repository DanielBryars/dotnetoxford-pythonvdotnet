Why Python Won ML

(and what C# can still do about it)

Daniel Bryars

If you start an ML project today… you pick Python. Why? This talk is about how that happened — and whether C# can claw back ground.

About Me

  • 20 years in .NET — C# by day, Python by night
  • Recently completed MSc in AI (heavy Python use)
I've been a .NET developer for 20 years, so C# is home. But my MSc in AI meant living in Python — Jupyter notebooks, PyTorch, NumPy — and I came to appreciate why it dominates ML.

Two Languages, Two Philosophies

Python

Guido van Rossum, 1991

  • Readability over speed
  • Scripting & teaching
  • Never intended for ML

C#

Anders Hejlsberg, 2000

  • Strong typing & tooling
  • Enterprise-focused
  • Closed source until 2014
Python was a teaching and scripting language — Guido wanted something between shell scripts and C. Nobody predicted it would dominate machine learning. C# was designed for building large, reliable systems. But being closed source and Windows-only for its first 14 years meant academia and the research community never touched it. By the time .NET went open source, Python had already won.

The Paradox

Python

  • Dynamic
  • ~100x slower
  • No type safety

C#

  • Strongly typed
  • JIT-compiled
  • Battle-tested tooling

Yet → Python dominates ML

By every traditional software engineering metric, C# should win. It's faster, safer, better tooled. But ML doesn't play by those rules — the bottleneck was never the language, it was the ecosystem.

What Actually Happened

  • 1991 PY Python released
  • 1995 PY NumPy predecessor (Numeric)
  • 2000 C# C# 1.0 (closed source, Windows-only)
  • 2001 PY SciPy & IPython (Fernando Pérez)
  • 2005 PY NumPy 1.0 (wraps Fortran BLAS/LAPACK)
  • 2007 PY scikit-learn begins
  • 2011 PY IPython Notebook launches
  • 2012 PY Deep learning explodes (AlexNet)

↓ more

Look at this timeline. By the time C# even existed, Python already had numerical computing locked down. By the time deep learning exploded in 2012, Python had a 20-year head start.

What Actually Happened (cont.)

  • 2014 C# .NET goes open source / cross-platform
  • 2014 PY Jupyter Project (from IPython)
  • 2015 PY TensorFlow released (Google)
  • 2016 PY PyTorch released (Facebook/Meta)
  • 2018 C# ML.NET 1.0
  • 2021 C# TorchSharp (PyTorch for .NET)
  • 2022 C# Generic Math / INumber<T> (.NET 7)
C# didn't go open source until 2014. ML.NET arrived in 2018 — six years after deep learning took off. Generic math with INumber<T> finally landed in .NET 7 in 2022. Too late to change the default, but not too late to be useful.

Python’s Unfair Advantage

  • NumPy — vectorised maths, Fortran BLAS/LAPACK core
  • PyTorch / TensorFlow — GPU-native, CUDA kernels
  • Jupyter — REPL-driven, visual, shareable

Python doesn’t do the maths — it orchestrates C, Fortran, and CUDA

↓ Jupyter vs LINQPad

This is the key insight. Python is slow — everyone knows that. But Python ML is fast because Python never does the heavy lifting. NumPy calls into BLAS and LAPACK — battle-hardened Fortran code from the 1970s. PyTorch dispatches to CUDA kernels. Python is just the steering wheel.

Jupyter vs LINQPad

Jupyter

  • Fernando Pérez, 2001 (as IPython)
  • Browser-based notebooks
  • Inline plots & rich output
  • Became the standard in academia

LINQPad

  • Joseph Albahari, 2007
  • Desktop scratchpad for .NET
  • Brilliant — but niche
  • No academic adoption

Same idea, different worlds. Jupyter reached researchers. LINQPad reached developers.

I remember being repulsed by Jupyter when I first used it — coming from C# and Visual Studio, it felt primitive. But the feedback loop — run a cell, see the plot, tweak — is addictive. LINQPad is arguably better as a tool, but Jupyter became the interface to ML. And doing a REPL in C# in 2010? Painful.

The Killer Property: Duck Typing

def f(x):
    return x*x + 2*x
f(3)              # int → 15
f(2.5)            # float → 11.25
f(1+2j)           # complex → (-2+6j)

import numpy as np
f(np.array([1,2,3]))   # ndarray → [3, 8, 15]

import torch
f(torch.tensor([1.,2.,3.]))  # GPU tensor

One function. Five domains. Zero changes.

↓ same thing in C#

This is the slide that should make the audience pause. One function — no generics, no overloads, no interfaces — and it works on integers, floats, complex numbers, arrays of a million elements, and GPU tensors. Duck typing means if it quacks like a number, Python will do maths on it.

The Same Thing in C#

// .NET 7+ with Generic Math
T F<T>(T x) where T : INumber<T>
{
    return x * x + (T.One + T.One) * x;
}

F(3);         // int → 15
F(2.5);       // double → 11.25
// Complex? Need a different overload.
// Arrays? Manual loop or LINQ.
// GPU tensors? Completely different API.

C# can do it — but each domain needs new code

C# got generic math in .NET 7 — INumber<T> — which is genuinely cool. But it only covers numeric types. Complex numbers don't implement INumber. Arrays need LINQ or loops. GPU tensors are a completely different world (TorchSharp). In Python, duck typing makes this free. In C#, every new domain is more code.

What C# Can Still Do About It

  • ML.NET — production inference, no Python dependency
  • TorchSharp — PyTorch bindings for .NET
  • ONNX Runtime — deploy any model, any framework
  • Semantic Kernel — AI orchestration in C#
  • Generic Math — INumber<T> (.NET 7+)

Train in Python. Ship in C#.

C# doesn't need to win training — that ship has sailed. But deployment? That's where type safety, performance, and tooling shine. ONNX Runtime lets you train a model in PyTorch and run it in a C# microservice. Semantic Kernel is Microsoft's play for AI-native C# apps. The smart play: use both.

Takeaway

Python won because it made
fast things easy

C# can win where it makes
safe things fast

Python owns experimentation — the REPL, the notebooks, the duck typing, the ecosystem. C# owns production — the type safety, the performance, the tooling. The real answer isn't "pick one" — it's knowing when to use which. Train in Python, ship in C#.

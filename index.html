Two Languages, Two Philosophies

Python (1991)

Guido van Rossum

  • Readability — “code is read more than written”
  • Glue language — shell scripts and C
  • A hobby project — Christmas 1989
  • Bundled with Linux — on every server by 2000

C# (2000)

Anders Hejlsberg

  • Java lawsuit — MS needed its own language
  • C++ + Delphi — created Delphi
  • Closed source — until 2014
  • Windows-only — no Linux until .NET Core
Python: Christmas 1989 hobby project. Guido wanted a "descendant of ABC that would appeal to Unix/C hackers" — a glue language for tasks too hard for shell scripts but too small for C. C#: after MS lost the Sun/Java lawsuit, Anders Hejlsberg (created Turbo Pascal and Delphi) designed C# combining C++, Java, and Delphi's component model for .NET. Closed source and Windows-only for 14 years meant no presence in academia.

By ~2000, Python shipped with major Linux distros — Red Hat, Debian — used for system tools and installers. Part of the OS. A PhD student could ssh into a server and run python3 script.py — no admin rights, no installs. SciPy: Matlab licences cost thousands per seat, per year, per toolbox. SciPy gave the same capabilities for free.

C# open sourced in 2014. ML.NET arrived in 2018 — six years after deep learning took off. Generic math with INumber<T> landed in .NET 7, 2022.

Python

  • Dynamic
  • ~100x slower
  • No type safety

C#

  • Strongly typed

Yet → Python dominates ML

By traditional software engineering metrics, C# scores higher — faster, type-safe, better tooling. But the ML ecosystem developed around Python.

Ecosystem Convergence

  • NumPy — Fortran BLAS/LAPACK under the hood
  • PyTorch / TensorFlow — CUDA kernels
  • Jupyter — interactive notebooks
Python delegates computation to compiled code. NumPy calls BLAS/LAPACK (Fortran, 1970s). PyTorch dispatches to CUDA kernels. One standard per job — everyone uses NumPy arrays, so matplotlib, SciPy, pandas, and PyTorch all interoperate without adapters. In C#, Math.NET has its own Matrix, ML.NET has DataView, TorchSharp has Tensor — different types at every boundary.

Everything speaks NumPy

import numpy as np
from scipy import signal
import matplotlib.pyplot as plt

t = np.linspace(0, 1, 1000)               # NumPy
noisy = np.sin(2*np.pi*5*t) + np.random.randn(len(t))
filtered = signal.filtfilt(*signal.butter(4, 15, fs=1000), noisy)  # SciPy ← NumPy
plt.plot(t, filtered)                      # Matplotlib ← NumPy

One array type. Every library accepts it. No adapters needed.

In C# you'd need Math.NET's Vector for numerics, a different type for ML.NET, another for TorchSharp — conversion code at every boundary. In Python, NumPy arrays are accepted and returned by SciPy, matplotlib, pandas, and PyTorch. Duck typing helps: if it has shape and supports arithmetic, it works.

Python doesn’t do the maths — it orchestrates C, Fortran, and CUDA

# @ calls into BLAS dgemm (Fortran) — not Python
C = A @ B                        # matrix multiply

# Broadcasting — no loops, no copies
prices = np.array([10, 20, 30])  # shape (3,)
tax    = np.array([[1.1],
                   [1.2]])       # shape (2, 1)
totals = prices * tax            # shape (2, 3) — automatic

# Vectorised operations — one call to C
distances = np.sqrt(np.sum((points - centre)**2, axis=1))

# Slicing — views, not copies
batch = images[0:32, :, :, :]    # zero-cost slice

Succinct and powerful — but can trip you up (print shapes early and often)

The @ operator maps to a BLAS dgemm call (Fortran). Broadcasting: multiply a (3,) vector by a (2,1) matrix, get a (2,3) result — no loops, no allocation. Vectorised sqrt across thousands of points is a single C call. Slicing returns views, not copies. The implicit shape rules can be confusing — printing .shape is the main debugging technique.

Duck Typing

Python

def f(x):
    return x*x + 2*x

f(3)                        # 15
f(2.5)                      # 11.25
f(1+2j)                     # (-2+6j)
f(np.array([1,2,3]))        # [3, 8, 15]
f(torch.tensor([1.,2.,3.])) # GPU tensor

C#

T F<T>(T x) where T : INumber<T>
{
    var two = T.One + T.One;
    return x * x + two * x;
}

F(3);       // int
F(2.5);     // double
// Complex?  won't compile
// Array?    need LINQ
// Tensor?   different API

Python dynamically resolves the * and + operators at runtime — if the type supports them, it just works

One function, no generics, no overloads, no interfaces — works on int, float, complex, ndarray, and GPU tensor. In C#, INumber covers int and double, but Complex doesn't implement it, arrays need LINQ, tensors use a separate API. Python resolves the operators dynamically at runtime.

Autograd — Calculus for Free

Python

import torch

x = torch.tensor(3.0, requires_grad=True)

y = x**2 + 2*x + 1
y.backward()

print(x.grad)  # 8.0

C#

using TorchSharp;

var x = torch.tensor(3.0f,
    requiresGrad: true);

var y = x.pow(2) + 2 * x + 1;
y.backward();

Console.WriteLine(
    x.grad()!.item<float>());

x**2 vs x.pow(2) • x.grad vs x.grad()!.item<float>() • Paper cuts add up

Both compute dy/dx = 2x + 2 = 8. Differences: Python has ** for power, C# needs .pow(). Python's .grad is a property, C#'s is a method returning nullable, needs ! operator, and .item needs a generic type parameter.

GPU Computing

Python

import torch

a = torch.randn(1000, 1000)
b = torch.randn(1000, 1000)
c = a @ b           # CPU

a, b = a.cuda(), b.cuda()
c = a @ b           # GPU

C#

using TorchSharp;

var a = torch.randn(1000, 1000);
var b = torch.randn(1000, 1000);
var c = a.mm(b);     // CPU

a = a.cuda();
b = b.cuda();
c = a.mm(b);         // GPU
TorchSharp mirrors PyTorch closely here. @ vs .mm() is minor. The setup friction is the difference: choosing the right native backend NuGet package (~800MB), matching CUDA toolkit version. Python: pip install torch.

Train a Neural Network (XOR)

Python — 12 lines

import torch
import torch.nn as nn

X = torch.tensor([[0,0],[0,1],
    [1,0],[1,1]], dtype=torch.float)
y = torch.tensor([[0],[1],
    [1],[0]], dtype=torch.float)

model = nn.Sequential(
    nn.Linear(2, 8), nn.ReLU(),
    nn.Linear(8, 1), nn.Sigmoid())
opt = torch.optim.Adam(
    model.parameters(), lr=0.01)

for _ in range(1000):
    loss = nn.functional \
        .binary_cross_entropy(
            model(X), y)
    opt.zero_grad()
    loss.backward()
    opt.step()

C# — same thing?

using TorchSharp;
using static TorchSharp.torch;
using static TorchSharp.torch.nn;

var X = torch.tensor(new float[,]
    {{0,0},{0,1},{1,0},{1,1}});
var y = torch.tensor(new float[,]
    {{0},{1},{1},{0}});

var model = Sequential(
    ("fc1", Linear(2, 8)),
    ("relu", ReLU()),
    ("fc2", Linear(8, 1)),
    ("sig", Sigmoid()));
var opt = torch.optim.Adam(
    model.parameters(),
    learningRate: 0.01);

for (int epoch = 0; epoch < 1000;
     epoch++) {
    using var pred =
        model.forward(X);
    using var loss = functional
        .binary_cross_entropy(
            pred, y);
    opt.zero_grad();
    loss.backward();
    opt.step();
}

Named tuples • using var for every tensor • model.forward(X) vs model(X) • learningRate: vs lr=

Both train the same XOR network. C# differences: named layer tuples, using var for tensor disposal (unmanaged memory), .forward() instead of calling the model directly, learningRate vs lr.
Lorenz attractor — three coupled differential equations. 15 lines of Python.

Lorenz Attractor

Python

import numpy as np
import matplotlib.pyplot as plt

dt, n = 0.01, 10000
sigma, beta, rho = 10, 8/3, 28
xyz = np.zeros((n, 3))
xyz[0] = [0.1, 0, 0]

for i in range(n - 1):
    x, y, z = xyz[i]
    xyz[i+1] = xyz[i] + dt*np.array([
        sigma*(y-x),
        x*(rho-z) - y,
        x*y - beta*z])

fig = plt.figure(facecolor="black")
ax = fig.add_subplot(projection="3d")
ax.scatter(*xyz.T,
    c=np.arange(n), cmap="plasma")

C#

const double dt = 0.01;
const int n = 10000;
const double sigma = 10,
    beta = 8.0/3, rho = 28;

var x = new double[n];
var y = new double[n];
var z = new double[n];
x[0] = 0.1;

for (int i = 0; i < n-1; i++) {
    x[i+1] = x[i]+dt*sigma*(y[i]-x[i]);
    y[i+1] = y[i]+dt*(x[i]*(rho-z[i])-y[i]);
    z[i+1] = z[i]+dt*(x[i]*y[i]-beta*z[i]);
}

// Plot? ScottPlot? OxyPlot?
// None do 3D scatter well.
// Export to CSV, open in Python.
The computation code is comparable. The difference is plotting — Python: three lines with matplotlib. C#: pick a plotting library, add the NuGet, configure it, and none handle 3D scatter well. Typical outcome: export to CSV and plot in Python.

Two Languages, Two Philosophies

Python (1991)

Guido van Rossum

  • Readability over speed
  • Scripting & teaching
  • Never intended for ML

C# (2000)

Anders Hejlsberg

  • Strong typing & tooling
  • Enterprise focused
  • Closed source until 2014
Python was a teaching and scripting language — Guido wanted something between shell scripts and C. Nobody predicted it would dominate machine learning. C# was designed for building large, reliable systems. But being closed source and Windows-only for its first 14 years meant academia and the research community never touched it. By the time .NET went open source, Python had already won.

Look at this timeline. By the time C# even existed, Python already had numerical computing locked down. SciPy was huge — Matlab licences cost thousands per seat, per year, per toolbox. SciPy gave academia the same capabilities for free. That alone drove massive adoption. By the time deep learning exploded in 2012, Python had a 20-year head start.

  • 2014 C# .NET goes open source / cross-platform
  • 2014 PY Jupyter Project (from IPython)
  • 2015 PY TensorFlow released (Google)
  • 2016 PY PyTorch released (Facebook/Meta)

C# didn't go open source until 2014. ML.NET arrived in 2018 — six years after deep learning took off. Generic math with INumber<T> finally landed in .NET 7 in 2022. Too late to change the default, but not too late to be useful.

Yet → Python dominates ML

By every traditional software engineering metric, C# should win. It's faster, safer, better tooled. But ML doesn't play by those rules — the bottleneck was never the language, it was the ecosystem.

Ecosystem Convergence

  • NumPy — Fortran BLAS/LAPACK under the hood
  • PyTorch / TensorFlow — CUDA kernels
  • Jupyter — interactive notebooks

One standard per job — everything speaks NumPy

This is the key insight. Python is slow — everyone knows that. But Python ML is fast because Python never does the heavy lifting. NumPy calls into BLAS and LAPACK — battle-hardened Fortran code from the 1970s. PyTorch dispatches to CUDA kernels. Python is just the steering wheel. And crucially — there's one standard per job. Everyone uses NumPy arrays. So matplotlib knows how to plot them. SciPy knows how to filter them. pandas knows how to store them. PyTorch can convert to and from them. The whole ecosystem converges on one array type. In C#, Math.NET has its own Matrix, ML.NET has DataView, TorchSharp has Tensor — nothing talks to anything else natively.

Python doesn’t do the maths — it orchestrates C, Fortran, and CUDA

# @ calls into BLAS dgemm (Fortran) — not Python
C = A @ B                        # matrix multiply

# Broadcasting — no loops, no copies
prices = np.array([10, 20, 30])  # shape (3,)
tax    = np.array([[1.1],
                   [1.2]])       # shape (2, 1)
totals = prices * tax            # shape (2, 3) — automatic

# Vectorised operations — one call to C
distances = np.sqrt(np.sum((points - centre)**2, axis=1))

# Slicing — views, not copies
batch = images[0:32, :, :, :]    # zero-cost slice

Succinct and powerful — but can trip you up (print shapes early and often)

The @ operator compiles down to a BLAS dgemm call — battle-tested Fortran from the 1970s. Broadcasting lets you multiply a (3,) vector by a (2,1) matrix and get a (2,3) result with no explicit loops or memory allocation. Vectorised sqrt across thousands of points is a single C call. Slicing gives you views, not copies, so it's zero-cost. This is incredibly powerful, but the implicit shape rules trip up beginners constantly — the number one debugging technique in NumPy is printing .shape everywhere.

I remember being repulsed by Jupyter when I first used it — coming from C# and Visual Studio, it felt primitive. But the feedback loop — run a cell, see the plot, tweak — is addictive. LINQPad is arguably better as a tool, but Jupyter became the interface to ML. And doing a REPL in C# in 2010? Painful.

Everything speaks NumPy

import numpy as np
from scipy import signal
import matplotlib.pyplot as plt

t = np.linspace(0, 1, 1000)               # NumPy
noisy = np.sin(2*np.pi*5*t) + np.random.randn(len(t))
filtered = signal.filtfilt(*signal.butter(4, 15, fs=1000), noisy)  # SciPy ← NumPy
plt.plot(t, filtered)                      # Matplotlib ← NumPy

One array type. Every library accepts it. No adapters needed.

In C# you'd need Math.NET's Vector for numerics, a different type for ML.NET, another for TorchSharp. Every boundary means conversion code. In Python, NumPy is the universal currency — SciPy, matplotlib, pandas, PyTorch all accept and return NumPy arrays. Duck typing makes this natural: if it has shape and supports arithmetic, it just works. This is the network effect at the library level.

Takeaway

Python won because it made
fast things easy

C# can win where it makes
safe things fast

Python owns experimentation — the REPL, the notebooks, the duck typing, the ecosystem. C# owns production — the type safety, the performance, the tooling. The real answer isn't "pick one" — it's knowing when to use which. Train in Python, ship in C#.

Autograd — Calculus for Free

Python

import torch

x = torch.tensor(3.0, requires_grad=True)

y = x**2 + 2*x + 1
y.backward()

print(x.grad)  # 8.0

C#

using TorchSharp;

var x = torch.tensor(3.0f,
    requiresGrad: true);

var y = x.pow(2) + 2 * x + 1;
y.backward();

Console.WriteLine(
    x.grad()!.item<float>());

x**2 vs x.pow(2) • x.grad vs x.grad()!.item<float>() • Paper cuts add up

The maths is identical — both compute dy/dx = 2x + 2 = 8. But look at the ergonomics. Python has operator overloading for power. C# needs .pow(). Python's .grad is a property. C#'s is a method that returns nullable, needs bang operator, and .item needs a generic type parameter. Each one is small. Together they make experimentation slower.

GPU Computing

Python

import torch

a = torch.randn(1000, 1000)
b = torch.randn(1000, 1000)
c = a @ b           # CPU

a, b = a.cuda(), b.cuda()
c = a @ b           # GPU

C#

using TorchSharp;

var a = torch.randn(1000, 1000);
var b = torch.randn(1000, 1000);
var c = a.mm(b);     // CPU

a = a.cuda();
b = b.cuda();
c = a.mm(b);         // GPU
This is actually TorchSharp's strongest showing — the API mirrors PyTorch almost exactly. The @ operator vs .mm() is minor. The real friction is setup: you need to pick the right native backend NuGet package, it's an 800MB download, and it has to match your CUDA toolkit version exactly. In Python: pip install torch — done.

Train a Neural Network (XOR)

Python — 12 lines

import torch
import torch.nn as nn

X = torch.tensor([[0,0],[0,1],
    [1,0],[1,1]], dtype=torch.float)
y = torch.tensor([[0],[1],
    [1],[0]], dtype=torch.float)

model = nn.Sequential(
    nn.Linear(2, 8), nn.ReLU(),
    nn.Linear(8, 1), nn.Sigmoid())
opt = torch.optim.Adam(
    model.parameters(), lr=0.01)

for _ in range(1000):
    loss = nn.functional \
        .binary_cross_entropy(
            model(X), y)
    opt.zero_grad()
    loss.backward()
    opt.step()

C# — same thing?

using TorchSharp;
using static TorchSharp.torch;
using static TorchSharp.torch.nn;

var X = torch.tensor(new float[,]
    {{0,0},{0,1},{1,0},{1,1}});
var y = torch.tensor(new float[,]
    {{0},{1},{1},{0}});

var model = Sequential(
    ("fc1", Linear(2, 8)),
    ("relu", ReLU()),
    ("fc2", Linear(8, 1)),
    ("sig", Sigmoid()));
var opt = torch.optim.Adam(
    model.parameters(),
    learningRate: 0.01);

for (int epoch = 0; epoch < 1000;
     epoch++) {
    using var pred =
        model.forward(X);
    using var loss = functional
        .binary_cross_entropy(
            pred, y);
    opt.zero_grad();
    loss.backward();
    opt.step();
}

Named tuples • using var for every tensor • model.forward(X) vs model(X) • learningRate: vs lr=

Both train the same XOR network. Both work. But the C# version has named layer tuples, using var for tensor disposal (every tensor is unmanaged memory), .forward() instead of calling the model directly, and the longer parameter name learningRate vs lr. None of these are deal-breakers — but in a research loop where you're iterating fast, every character counts.
Lorenz attractor — a chaotic system of three coupled differential equations. 15 lines of Python, one stunning plot.

Lorenz Attractor

Python

import numpy as np
import matplotlib.pyplot as plt

dt, n = 0.01, 10000
sigma, beta, rho = 10, 8/3, 28
xyz = np.zeros((n, 3))
xyz[0] = [0.1, 0, 0]

for i in range(n - 1):
    x, y, z = xyz[i]
    xyz[i+1] = xyz[i] + dt*np.array([
        sigma*(y-x),
        x*(rho-z) - y,
        x*y - beta*z])

fig = plt.figure(facecolor="black")
ax = fig.add_subplot(projection="3d")
ax.scatter(*xyz.T,
    c=np.arange(n), cmap="plasma")

C#

const double dt = 0.01;
const int n = 10000;
const double sigma = 10,
    beta = 8.0/3, rho = 28;

var x = new double[n];
var y = new double[n];
var z = new double[n];
x[0] = 0.1;

for (int i = 0; i < n-1; i++) {
    x[i+1] = x[i]+dt*sigma*(y[i]-x[i]);
    y[i+1] = y[i]+dt*(x[i]*(rho-z[i])-y[i]);
    z[i+1] = z[i]+dt*(x[i]*y[i]-beta*z[i]);
}

// Plot? ScottPlot? OxyPlot?
// None do 3D scatter well.
// Export to CSV, open in Python.
The maths is comparable — C# even looks reasonable for the computation. But then you need to plot it. Python: three lines. C#: pick a library, add the NuGet, write 20 lines of config, realise it doesn't do 3D scatter well, export to CSV, and open it in Python anyway. That last line always gets a laugh.

Ecosystem — Everyone Speaks NumPy

Python

import numpy as np
from scipy import signal
import matplotlib.pyplot as plt

t = np.linspace(0, 1, 1000)
noisy = np.sin(2*np.pi*5*t) \
      + np.random.randn(len(t))

b, a = signal.butter(4, 15, fs=1000)
filtered = signal.filtfilt(b, a, noisy)

plt.plot(t, filtered)
plt.show()

This is the ecosystem convergence slide. NumPy array goes into SciPy, comes back as NumPy, goes into matplotlib. Zero conversion. In C# you'd need three different libraries with three different types — Math.NET for numerics, something else for filtering, something else for plotting — and adapters between all of them. Or just call Python.
